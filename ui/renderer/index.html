两端填写参考

  • 推流端（desktop-服务端模式）：服务地址填 run/runtime.json 里的 gateway.url，app 与 stream 与本页创建流保持一致。
  • 拉流端（desktop-服务端拉流）：服务地址同上，填写相同 app 与 stream，优先协议建议 HTTP-FLV。
  • 若跨机器部署，请放行 runtime.json 中 gateway 端口及 RTMP/HTTP 端口。

提示：推流-服务端模式下，分辨率和帧率请勿设置过高，防止服务端负载过大影响拉流稳定性。建议不要超过 1920x1080 & 60fps。

提示

完整链路（服务端模式）是这样：

1) 推流端（desktop）先向 gateway 发 HTTP API 请求

类似 POST /api/v1/streams/start

这是“控制面”协议：HTTP/JSON（不是媒体流）

2) gateway 返回会话信息

包括 publishRtmp、playHttpFlv、id 等

这里已经把“该推到哪、该拉哪个流”编排好了

3) 推流端再把媒体流推到 publishRtmp

这一段是“数据面”协议：RTMP

也就是编码后音视频真正上传到服务端的通道

4) 拉流端按需要拉取播放地址

常见是 playHttpFlv（HTTP-FLV），也可用 RTMP/WebRTC（看服务端配置和客户端能力）

这一段是服务端分发给观众/播放器的协议

此 UI 仅管理地址与状态，媒体分发由 service 内置 ZLMediaKit 内核承担。
