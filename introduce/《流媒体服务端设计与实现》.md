# 流媒体服务端设计与实现

## 一、ZLMediaKit 集成

本项目选用 ZLMediaKit 作为核心流媒体引擎，负责所有媒体数据的接收、转码与分发工作。ZLMediaKit 是一个高性能的 C++ 流媒体服务器，具备完备的协议支持能力，覆盖 RTMP、RTSP、HLS、HTTP-FLV、WebRTC、SRT 等多种流媒体协议。

集成方式上，我们没有将 ZLMediaKit 作为源码编译到项目中，而是将其预编译为二进制可执行文件，放置在 `3rd/zlm` 目录下，分别提供 Windows 版本的 `MediaServer.exe` 和 Linux 版本的 `MediaServer`。每个平台目录下同时附带对应的 `config.ini` 配置文件模板。

系统启动时，各启动器均将 ZLMediaKit 作为独立的子进程启动。流程为：动态检测可用端口 → 将配置模板中的端口替换为实际端口 → 生成运行时配置文件 → 以子进程方式启动 MediaServer。以下是 Kernel Console（Go 语言启动器）中的核心启动逻辑：

```go
// kernel-console/main.go —— 动态端口分配、配置补丁、子进程启动
rtmpPort := getFreePort(1935)
httpPort := getFreePort(8080)
httpsPort := getFreePort(8443)
rtspPort := 0
if runtime.GOOS == "linux" {
    rtspPort = getFreePort(8554)
}
patchConfig(layout.ZLMCfg, layout.ZLMRuntimeCfg, rtmpPort, httpPort, httpsPort, rtspPort)

zlmCmd := exec.Command(layout.ZLMExe, "-c", layout.ZLMRuntimeCfg)
zlmCmd.Dir = filepath.Dir(layout.ZLMExe)
zlmCmd.Stdout = zlmOut
zlmCmd.Stderr = zlmErr
zlmCmd.Start()
```

ZLMediaKit 与 Go Gateway 之间不通过进程间通信或 API 回调交互，而是通过环境变量传递端口信息。启动器在启动 Gateway 子进程时注入 `ZLM_RTMP_PORT`、`ZLM_HTTP_PORT` 等环境变量，Gateway 启动时通过 `getEnvInt` 函数读取这些变量来生成正确的播放地址。ZLMediaKit 的 Hook 机制未启用，所有流的管理逻辑由 Gateway 独立完成。

配置层面，ZLMediaKit 开启了全部协议转换能力，所有协议的按需转换均被关闭，流一推入即主动生成所有协议输出，保证首帧低延迟播放。关键配置如下：

```ini
[protocol]
modify_stamp=2
enable_audio=1
add_mute_audio=1
enable_hls=1
enable_rtsp=1
enable_rtmp=1
enable_ts=1
enable_fmp4=1
hls_demand=0
rtsp_demand=0
rtmp_demand=0
ts_demand=0
fmp4_demand=0

[hls]
segDur=2
segNum=3
segRetain=5
```

## 二、Go Gateway

Go Gateway 是整个系统的控制面核心，负责对外提供 HTTP API 以便客户端创建、查询和管理流媒体会话。Gateway 使用 Go 1.22 标准库开发，不依赖任何第三方框架，代码集中在一个约 860 行的 `main.go` 文件中。

核心数据模型和路由注册如下：

```go
// gateway/main.go —— 数据模型、路由注册与 CORS 中间件
type streamRecord struct {
    ID            string            `json:"id"`
    App           string            `json:"app"`
    Stream        string            `json:"stream"`
    PublishHost   string            `json:"publishHost"`
    ServiceMode   string            `json:"serviceMode"`
    CreatedAt     time.Time         `json:"createdAt"`
    Status        streamStatus      `json:"status"`
    PlayURLs      map[string]string `json:"playUrls"`
}

var (
    mu       sync.RWMutex
    streams  = map[string]*streamRecord{}
    rtmpPort = getEnvInt("ZLM_RTMP_PORT", 1935)
    httpPort = getEnvInt("ZLM_HTTP_PORT", 8080)
)

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/healthz", healthzHandler)
    mux.HandleFunc("/api/v1/streams/start", startHandler)
    mux.HandleFunc("/api/v1/streams/resolve", resolveHandler)
    mux.HandleFunc("/api/v1/streams/", streamByIDHandler)
    mux.HandleFunc("/api/v1/streams", listHandler)
    mux.HandleFunc("/api/v1/debug/logs", logsHandler)
    mux.HandleFunc("/api/v1/debug/network", networkDebugHandler)
    http.ListenAndServe(":"+strconv.Itoa(gatewayPort), corsMiddleware(mux))
}
```

Gateway 的状态存储采用内存映射方式，使用 `sync.RWMutex` 保护。对外暴露的核心接口包括：`GET /healthz`（健康检查）、`POST /api/v1/streams/start`（创建流会话）、`GET /api/v1/streams/resolve`（按 app/stream 解析播放地址）、`GET /api/v1/streams`（流列表）、`POST /api/v1/streams/<id>/stop`（停止流）、`GET /api/v1/streams/<id>/status`（查询状态），以及调试接口 `GET /api/v1/debug/logs` 和 `GET /api/v1/debug/network`。所有接口配置了允许任意来源的 CORS 中间件。

Gateway 设计了三种服务模式：直连模式（direct）仅生成 RTMP 地址；广播模式（broadcast）和 HTTP-FLV 模式（httpflv）则同时生成 RTMP、HTTP-FLV 和 HLS 播放地址。播放地址的生成过程中，Gateway 实现了多级主机地址选择：公网代理配置优先 → 启动阶段选定的 IP → Linux 下路由优选和/24子网匹配 → X-Forwarded-Host 头 → 本机 IP。关键代码如下：

```go
// gateway/main.go —— 主机选择（多级回退 + Linux 路由优选）
func publishHostForResponse(r *http.Request) string {
    if getPublicProxyEnabled() { return getPublicHostOverride() }
    if strings.TrimSpace(selectedPublishHost) != "" { return selectedPublishHost }
    return responseHostFallback(r)
}

func pickLinuxReachableHost(r *http.Request) string {
    candidates := localIPv4Candidates()
    clientIP := requestClientIP(r)
    // 通过 UDP 拨号获取路由优选的本机 IP
    if byRoute := routePreferredLocalIP(clientIP); byRoute != "" && isUsableIPv4(byRoute) {
        if canReachLocalService(byRoute) { return byRoute }
    }
    // 同 /24 子网的候选地址优先
    sameSubnet := filterBySubnet(candidates, clientIP)
    if host := pickRoundRobinReachable(sameSubnet); host != "" { return host }
    return pickRoundRobinReachable(candidates)
}
```

## 三、流编排

流编排采用基于会话的管理模型，围绕"创建会话—生成地址—解析查询—状态管理"四个环节展开。Gateway 仅负责控制面的会话管理，不参与数据平面的媒体处理。

`POST /api/v1/streams/start` 接收应用名、流名、服务模式等参数，首先进行幂等检查——相同 app+stream 的运行中会话直接返回已有结果；不存在则生成 16 位十六进制随机 ID，创建流记录存入内存映射，调用 `buildPlayURLs` 生成播放地址，返回 JSON 响应。客户端拿到 RTMP 推流地址后自行向 ZLMediaKit 推流。

```go
// gateway/main.go —— 流创建（幂等检查 + 会话管理）与地址生成
func startHandler(w http.ResponseWriter, r *http.Request) {
    var req startRequest
    json.NewDecoder(r.Body).Decode(&req)
    // 幂等检查
    mu.RLock()
    for _, rec := range streams {
        if rec.App == app && rec.Stream == stream && rec.Status == statusRunning {
            // 返回已有会话
            mu.RUnlock()
            writeJSON(w, http.StatusOK, resp)
            return
        }
    }
    mu.RUnlock()
    // 创建新会话
    id, _ := randomID()
    rec := &streamRecord{ID: id, App: app, Stream: stream, Status: statusRunning, CreatedAt: time.Now()}
    rec.PlayURLs = buildPlayURLs(host, rec.App, rec.Stream, rec.ServiceMode)
    mu.Lock()
    streams[id] = rec
    mu.Unlock()
    // 返回会话信息
}

func buildPlayURLs(host, app, stream, mode string) map[string]string {
    urls := map[string]string{
        "rtmp": "rtmp://" + host + ":" + strconv.Itoa(rtmpPort) + "/" + app + "/" + stream,
    }
    if mode == "httpflv" || mode == "broadcast" {
        urls["httpFlv"] = "http://" + host + ":" + strconv.Itoa(httpPort) + "/" + app + "/" + stream + ".flv"
        urls["hls"] = "http://" + host + ":" + strconv.Itoa(httpPort) + "/" + app + "/" + stream + "/hls.m3u8"
    }
    return urls
}
```

`GET /api/v1/streams/resolve?app=X&stream=Y` 允许播放端仅凭业务层的 app 和 stream 即可查找到最近的匹配会话及其播放地址，无需维护会话 ID。流状态仅有 running 和 stopped 两种，停止操作仅修改 Gateway 中的状态标记，ZLM 的流生命周期由其内部独立管理。

## 四、协议转换

协议转换完全由 ZLMediaKit 内置引擎完成，Gateway 不参与媒体层面的处理。ZLMediaKit 开启了 RTMP/FLV（支持 H.265/VP9 Enhanced RTMP）、RTSP（直接代理）、HLS（MPEG-TS，2秒切片）、HTTP-TS/WS-TS、HTTP-FMP4/WS-FMP4、WebRTC（STUN/TURN）以及 SRT 等全部协议。所有按需转换关闭，推流进入后立即主动生成全部协议输出，首帧播放无需等待。

Gateway 通过 `buildPlayURLs` 按固定 URL 模板生成地址，与 ZLMediaKit 的访问路径规则保持一致：
- RTMP: `rtmp://<host>:<port>/<app>/<stream>`
- HTTP-FLV: `http://<host>:<port>/<app>/<stream>.flv`
- HLS: `http://<host>:<port>/<app>/<stream>/hls.m3u8`

## 五、健康检查

系统健康检查分为六个层次，形成完整的监控链路。

第一层，Gateway 暴露 `GET /healthz` 端点，返回 `{"status":"ok"}`，是所有上层组件判断服务就绪的唯一依据。

第二层，启动阶段健康轮询——所有启动器以 200ms 间隔轮询 `/healthz`，超时后换端口重试，最多 8 次：

```go
// kernel-console/main.go —— 健康轮询与重试
func waitGatewayHealthy(port int, timeout time.Duration) bool {
    deadline := time.Now().Add(timeout)
    for time.Now().Before(deadline) {
        resp, err := http.Get(fmt.Sprintf("http://127.0.0.1:%d/healthz", port))
        if err == nil && resp.StatusCode == 200 { return true }
        time.Sleep(200 * time.Millisecond)
    }
    return false
}
// 最多重试 8 次，每次换空闲端口
for i := 0; i < 8; i++ {
    candidate := getFreePort(preferred)
    // ... 启动 gateway ...
    if waitGatewayHealthy(candidate, 5*time.Second) { break }
    cmd.Process.Kill()
}
```

第三层，Electron 托管模式在启动 Gateway 之前通过 `waitTcpPort` 以 TCP 连接探测 ZLMediaKit 的 RTMP 端口（8 秒超时），确保媒体引擎先于 Gateway 就绪。

第四层，Gateway 内部通过 `tcpReachable` 对候选 IP 的 Gateway 端口和 RTMP 端口执行 300ms 超时的 TCP 可达性检测，并在启动阶段通过 `canBindLocalIP` 验证候选 IP 的可绑定性。

第五层，所有启动器在启动成功后写入 `run/runtime.json`，包含 Gateway URL、各端口号、启动时间戳等。Electron UI 通过读取该文件自动获取 Gateway 地址，无需手动配置：

```go
// kernel-console/main.go —— runtime.json 结构体
type runtimeInfo struct {
    StartedAt string `json:"startedAt"`
    Gateway   struct {
        URL  string `json:"url"`
        Host string `json:"host"`
        Port int    `json:"port"`
    } `json:"gateway"`
    ZLM struct {
        RTMPPort  int `json:"rtmpPort"`
        HTTPPort  int `json:"httpPort"`
        HTTPSPort int `json:"httpsPort"`
    } `json:"zlm"`
}
```

第六层，调试诊断接口 `GET /api/v1/debug/network` 返回当前主机、端口、OS、代理状态等信息，`GET /api/v1/debug/logs` 支持按 limit 参数返回各组件的日志尾部内容，方便问题排查。

## 六、跨平台部署

本项目完整支持 Windows 和 Linux x64 两大平台，从编译构建到打包发布形成了自动化的跨平台交付流水线。

Kernel Console 通过 `detectLayout` 函数自动适配不同平台的目录结构、二进制名称和 ZLM 路径：

```go
// kernel-console/main.go —— 跨平台目录检测
func detectLayout() (serviceLayout, error) {
    // 搜索 exe 同级、上级、上上级目录，找到服务根目录
    isWin := runtime.GOOS == "windows"
    zlmDir := filepath.Join(zlmBase, "windows")
    if !isWin {
        // Linux 下兼容 linux / Linux 两种大小写
        if _, err := os.Stat(filepath.Join(zlmBase, "linux")); err == nil {
            zlmDir = filepath.Join(zlmBase, "linux")
        } else if _, err := os.Stat(filepath.Join(zlmBase, "Linux")); err == nil {
            zlmDir = filepath.Join(zlmBase, "Linux")
        }
    }
    gatewayName := "gateway"
    zlmExeName := "MediaServer"
    if isWin { gatewayName = "gateway.exe"; zlmExeName = "MediaServer.exe" }
    // 组装各组件路径 ...
}
```

构建流水线按平台分离：`build-win-package.ps1` 负责 `go build` 编译 Gateway/Kernel Console、`npm install && npm run dist:win` 打包 Windows 便携版；`build-linux-package.sh` 执行相同流程并通过 electron-builder 输出 AppImage，同时配置了 `ELECTRON_MIRROR` 以加速国内网络环境下的 Electron 二进制下载。最终的发布组装脚本将产物收集到 `release/` 目录，拆分为两个分发包——便携版 UI 包（含 Electron 界面）和便携版内核包（纯命令行），并对必需文件进行完整性校验。

停止流程通过先发 SIGINT 信号、等待 1.5 秒后强制 kill 的方式实现优雅停止。Linux 平台额外开启 RTSP 端口（默认 8554 避免 root 权限需求），并通过 `--no-sandbox` 包装脚本规避 Chromium 沙箱的 root 权限限制。

Electron UI 通过 `getPlatformRuntimeLayout` 获取平台差异配置（二进制名称、脚本扩展名和运行器等），支持托管和非托管双模式。托管模式下 Electron 直接管理 ZLM 和 Gateway 的子进程生命周期——动态端口分配、配置文件生成、启动、等待端口就绪、健康检查重试、写入 `runtime.json`——与 Kernel Console 和脚本启动器逻辑一致；非托管模式下 UI 作为纯前端控制台接入已运行的服务后端。UI 通过 IPC 通道 `service:getDefaultGatewayUrl` 自动读取 `runtime.json` 获取 Gateway 地址，实现开箱即用。
