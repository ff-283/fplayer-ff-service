# 技术文档：service 实现说明

本文档面向开发者，详细说明 `fplayer-ff-service` 的实现方式与关键设计。

## 1. 目标与边界

### 1.1 目标

- 将媒体分发能力从 desktop 解耦到独立 service
- desktop 与 service 仅通过网络协议通信
- 支持会话创建、地址编排、服务端分发（RTMP/HTTP-FLV）

### 1.2 非目标（当前版本）

- 不做持久化数据库
- 不做强鉴权/多租户
- 不做转码编排与资源调度

## 2. 总体架构

service 由三层组成：

1. **媒体内核层（ZLMediaKit）**
   - 负责接收 RTMP 推流
   - 负责对外提供 HTTP-FLV / RTMP 拉流
2. **控制层（gateway，Go）**
   - 提供 HTTP API
   - 负责会话元数据管理
   - 负责生成可用推拉流地址
3. **管理层（Electron UI）**
   - 仅作会话操作与状态展示
   - 不直接承载媒体分发

## 3. 模块与目录

- `3rd/zlm/windows/`：ZLM 可执行及配置
- `gateway/main.go`：网关 API 与会话内存存储
- `ui/`：控制台界面
- `scripts/start-all.ps1`：统一启动入口
- `scripts/stop-all.ps1`：统一停止入口
- `run/runtime.json`：运行时端口与端点快照
- `logs/*`：子进程日志

## 4. 启动链路实现

当前有两条启动链路：

### 4.1 开发态（脚本编排）

`scripts/start-all.ps1` 的职责：

1. 校验依赖（`MediaServer.exe`，以及按启动路径校验 `gateway.exe` 或 `go`，UI 路径才校验 `npm`）
2. 自动探测空闲端口（RTMP/HTTP/HTTPS + gateway）
3. 基于模板 `config.ini` 生成运行时配置 `run/zlm.runtime.ini`
4. 启动 ZLM / gateway / UI 子进程
5. gateway 健康检查失败时自动换端口重试
6. 写入 pid 文件与 `run/runtime.json`
7. 启动子进程默认隐藏控制台窗口（避免弹出黑框）

### 4.2 发布态（Electron 内置编排）

打包后默认不依赖手工执行脚本，`ui/main.js` 在主进程内完成：

1. 端口探测
2. 启动 ZLM 与 gateway（`spawn`）
3. 健康检查（`/healthz`）
4. 写入 `run/runtime.json` 与 `run/*.pid`
5. 退出时停止子进程并清理运行时文件

这保证了“用户双击 exe 即可运行”。

发布态支持两种进程入口：

- `FPlayerFFService.exe`：UI 模式（创建窗口 + 编排服务核心）
- `FPlayerFFServiceKernel.exe`：内核模式（不创建窗口，启动即服务，命令行打印状态）

内核模式通过可执行文件名包含 `kernel` 自动识别，也兼容 `--kernel` 参数与 `SERVICE_KERNEL_MODE=1` 环境变量。

`scripts/stop-all.ps1` 通过 pid 文件逆序停止进程并清理运行时文件。

## 5. gateway 设计

### 5.1 进程模型

- 单进程 HTTP 服务（端口由启动链路动态分配，优先尝试 `9000`）
- 内存存储会话（`map[string]*streamRecord` + `sync.RWMutex`）

### 5.2 API

- `POST /api/v1/streams/start`
- `GET /api/v1/streams/{id}/status`
- `POST /api/v1/streams/{id}/stop`
- `GET /api/v1/streams`
- `GET /healthz`

### 5.3 关键数据结构

- `streamRecord`
  - `id/app/stream/status/serviceMode`
  - `publisherMeta/sourceMeta`
  - `playUrls`

### 5.4 地址生成策略

- 运行时从环境变量读取 ZLM 端口：
  - `ZLM_RTMP_PORT`
  - `ZLM_HTTP_PORT`
- `start/status` 统一返回：
  - `publishRtmp`
  - `playHttpFlv`
  - `playUrls`

## 6. desktop 协同协议

### 6.1 推流侧（服务端模式）

1. desktop 调 `POST /api/v1/streams/start`
2. service 返回 `publishRtmp`
3. desktop 将媒体流推到 `publishRtmp`

### 6.2 拉流侧（服务端模式）

1. 输入 `streamId`
2. desktop 调 `GET /api/v1/streams/{id}/status`
3. 选取 `playHttpFlv` 或 `playUrls.rtmp` 作为拉流地址

## 7. 两种业务模式的实现差异

### 7.1 P2P 模式

- 地址由 desktop 本地拼接
- service 不参与会话创建

### 7.2 服务端模式

- 地址由 service 动态下发
- desktop 只维护业务参数（服务地址/app/stream/模式）

## 8. 配置与端口

- gateway 端口优先：`9000`（冲突时自动切换）
- ZLM 端口动态：启动时自动分配
- desktop 最终应使用：`run/runtime.json` 中 `gateway.url`
- 实际端口来源：`run/runtime.json`

## 9. 失败与恢复

### 9.1 gateway 不可用

表现：UI 报 `Failed to fetch`

处理：

1. 以 `run/runtime.json` 的 `gateway.url` 检查 `healthz`
2. 查看 `logs/gateway.err.log`
3. 重启 `start-all.ps1`

### 9.2 UI 启动失败

典型原因：electron 依赖下载失败

处理：

1. `cd ui && npm install`
2. 必要时使用镜像源重装
3. 再执行 `start-all.ps1`

## 10. 可扩展方向

- 持久化会话（Redis/MySQL）
- 鉴权（publisher token / viewer token）
- 统一公网地址配置（`PUBLIC_HOST`）
- 多节点路由与健康探测
- 会话统计与观众计数
