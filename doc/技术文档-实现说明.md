# 技术文档：service 实现说明

本文档面向开发者，详细说明 `fplayer-ff-service` 的实现方式与关键设计。

## 1. 目标与边界

### 1.1 目标

- 将媒体分发能力从 desktop 解耦到独立 service
- desktop 与 service 仅通过网络协议通信
- 支持会话创建、地址编排、服务端分发（RTMP/HTTP-FLV）

### 1.2 非目标（当前版本）

- 不做持久化数据库
- 不做强鉴权/多租户
- 不做转码编排与资源调度

## 2. 总体架构

service 由四层组成：

1. **媒体内核层（ZLMediaKit）**
   - 负责接收 RTMP 推流
   - 负责对外提供 HTTP-FLV / RTMP 拉流
2. **控制层（gateway，Go）**
   - 提供 HTTP API
   - 负责会话元数据管理
   - 负责生成可用推拉流地址
3. **管理层（Electron UI）**
   - 仅作会话操作与状态展示
   - 不直接承载媒体分发
4. **控制台入口层（kernel-console，Go）**
   - 独立控制台启动器
   - 负责前台驻留、打印可复制链接、接收 `Ctrl+C` 停止

## 3. 模块与目录

- `3rd/zlm/windows/`：ZLM 可执行及配置
- `gateway/main.go`：网关 API 与会话内存存储
- `ui/`：控制台界面
- `kernel-console/`：独立控制台内核启动器
- `scripts/start-all.ps1`：统一启动入口
- `scripts/stop-all.ps1`：统一停止入口
- `run/runtime.json`：运行时端口与端点快照
- `logs/*`：子进程日志

## 4. 启动链路实现

当前有三条启动链路：

### 4.1 开发态（脚本编排）

`scripts/start-all.ps1` 的职责：

1. 校验依赖（`MediaServer.exe`，以及按启动路径校验 `gateway.exe` 或 `go`，UI 路径才校验 `npm`）
2. 自动探测空闲端口（RTMP/HTTP/HTTPS + gateway）
3. 基于模板 `config.ini` 生成运行时配置 `run/zlm.runtime.ini`
4. 启动 ZLM / gateway / UI 子进程
5. gateway 健康检查失败时自动换端口重试
6. 写入 pid 文件与 `run/runtime.json`
7. 启动子进程默认隐藏控制台窗口（避免弹出黑框）

### 4.2 发布态 UI（Electron 内置编排）

不依赖手工执行脚本，`ui/main.js` 在主进程内完成：

1. 打开窗口时**不自动启动服务**（`coreStartupStatus` 初始为 `idle`）
2. 用户点击「创建并开始」触发 `startServiceCore` IPC 启动链路
3. 端口探测
4. 启动 ZLM（`spawn`）→ **TCP 端口健康检查**（`waitTcpPort` 轮询 RTMP 端口，10s 超时；失败则 kill ZLM 并抛错）
5. ZLM 就绪后启动 gateway（可传入 UI 输入的优先端口号）
6. gateway 健康检查（`/healthz`，自动换端口重试最多 8 次）
7. 写入 `run/runtime.json` 与 `run/*.pid`
8. 启动成功后 IPC 回写 Gateway URL 到 UI 输入框（`refreshGatewayUrl`）
9. 窗口关闭或退出时停止子进程并清理运行时文件

用户双击 exe 打开界面，点击按钮才启动服务。

### 4.3 发布态 Kernel（Go 控制台启动器）

`kernel-console` 直接在 Go 代码中完成：

1. 端口探测
2. 生成 `run/zlm.runtime.ini`
3. 启动 `MediaServer.exe` 与 `gateway.exe`
4. 健康检查（`/healthz`）
5. 写入 `run/runtime.json`
6. 前台驻留并监听 `Ctrl+C`
7. 停止时优雅中断，超时后强制结束子进程

对应可执行文件：

- `FPlayerFFService.exe`：UI 模式（Electron）
- `FPlayerFFServiceKernel.exe`：控制台 Kernel 模式（Go）

`scripts/stop-all.ps1` 通过 pid 文件逆序停止进程并清理运行时文件。

## 5. gateway 设计

### 5.1 进程模型

- 单进程 HTTP 服务，端口由启动链路动态分配
- 端口优先级：UI 输入框指定端口 > 9000（默认）> 操作系统随机分配
- 内存存储会话（`map[string]*streamRecord` + `sync.RWMutex`）

### 5.2 API

- `POST /api/v1/streams/start` — **幂等创建**：若同 `app`+`stream` 已有 running 记录，直接返回已有流（200），不创建新记录
- `GET /api/v1/streams/{id}/status`
- `POST /api/v1/streams/{id}/stop`
- `GET /api/v1/streams`
- `GET /healthz`

### 5.3 关键数据结构

- `streamRecord`
  - `id/app/stream/status/serviceMode`
  - `publisherMeta/sourceMeta`
  - `playUrls`

### 5.4 地址生成策略

- 运行时从环境变量读取 ZLM 端口：
  - `ZLM_RTMP_PORT`
  - `ZLM_HTTP_PORT`
- `start/status` 统一返回：
  - `publishRtmp`
  - `playHttpFlv`
  - `playUrls`

## 6. desktop 协同协议

### 6.1 推流侧（服务端模式）

1. desktop 调 `POST /api/v1/streams/start`
2. service 返回 `publishRtmp`
3. desktop 将媒体流推到 `publishRtmp`

### 6.2 拉流侧（服务端模式）

1. 输入 `streamId`
2. desktop 调 `GET /api/v1/streams/{id}/status`
3. 选取 `playHttpFlv` 或 `playUrls.rtmp` 作为拉流地址

## 7. 两种业务模式的实现差异

### 7.1 P2P 模式

- 地址由 desktop 本地拼接
- service 不参与会话创建

### 7.2 服务端模式

- 地址由 service 动态下发
- desktop 只维护业务参数（服务地址/app/stream/模式）

## 8. 配置与端口

- gateway 端口优先：UI 指定 > `9000` > 随机（冲突时自动切换）
- ZLM 端口动态：启动时自动分配
- desktop 最终应使用：`run/runtime.json` 中 `gateway.url`
- 实际端口来源：`run/runtime.json`

## 9. 失败与恢复

### 9.1 gateway 不可用

表现：UI 报「无法连接网关，请检查 Gateway URL 是否正确或服务是否已启动」

处理：

1. 以 `run/runtime.json` 的 `gateway.url` 检查 `healthz`
2. 查看 `logs/gateway.err.log`
3. 点击「创建并开始」重新启动服务

### 9.2 ZLM 启动失败

表现：UI 状态显示「ZLM 启动超时，请查看日志」

处理：

1. 检查 `logs/zlm.err.log` 中是否有端口被占用、权限不足等错误
2. 确认 `3rd/zlm/` 下可执行文件存在且可执行
3. 点击「创建并开始」重试

### 9.3 UI 启动失败

典型原因：electron 依赖下载失败

处理：

1. `cd ui && npm install`
2. 必要时使用镜像源重装
3. 再执行 `start-all.ps1`

## 10. UI 特性（新增）

- **按需启动**：Electron 窗口打开后不自动启服务，点击「创建并开始」才拉起 ZLM + gateway
- **端口可指定**：`gatewayUrl` 输入框支持指定端口（如 `http://192.168.1.5:9020`），启动时优先使用
- **深浅色主题**：右上角切换按钮，`localStorage` 持久化，基于 Cohere 设计系统
- **幂等创建**：同名 app+stream 重复请求不报错，直接返回已有流

## 11. 可扩展方向

- 持久化会话（Redis/MySQL）
- 鉴权（publisher token / viewer token）
- 统一公网地址配置（`PUBLIC_HOST`）
- 多节点路由与健康探测
- 会话统计与观众计数
