# 服务端部署与使用说明

本文档说明如何将 `fplayer-ff-service` 作为独立软件部署，并与 `desktop` 通过网络协同。

## 1. 架构说明

- `desktop` 与 `service` 只通过网络通信。
- `service` 内部包含：
  - `ZLMediaKit`（媒体分发内核）
  - `gateway`（HTTP API，优先 `9000`，端口冲突时自动切换）
  - `ui`（可选管理界面）

典型流程：

1. desktop 调 `service` 的 `POST /api/v1/streams/start` 创建会话
2. service 返回 `publishRtmp` 和 `playHttpFlv`
3. desktop 向 `publishRtmp` 推流
4. 播放端从 `playHttpFlv` 拉流，或按 `streamId` 调 `status` 自动解析

## 2. 目录准备

确保以下目录存在：

- `3rd/zlm/windows/MediaServer.exe`
- `3rd/zlm/windows/config.ini`
- `gateway/`
- `ui/`
- `scripts/start-all.ps1`
- `scripts/stop-all.ps1`

## 3. 命令行启动（Windows）

在 `fplayer-ff-service` 根目录执行：

```powershell
.\scripts\start-all.ps1
```

脚本会自动完成：

- 启动 ZLM
- 启动 gateway（优先 `9000`，冲突自动切换）
- 启动 UI
- 自动选择空闲 RTMP/HTTP-FLV 端口
- 写入运行时信息到 `run/runtime.json`
- 默认隐藏子进程控制台窗口（避免弹出多个黑框）

### 查看运行状态

1) 网关健康检查：

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:<gateway-port>/healthz
```

2) 查看运行时端口：

```powershell
Get-Content .\run\runtime.json
```

3) 查看日志：

- `logs/zlm.log`
- `logs/gateway.log`
- `logs/ui.log`
- `logs/*.err.log`

## 3.1 发布包启动（Windows，推荐）

若你使用发布包，可按模式选择：

- UI 模式：`FPlayerFFService.exe`
- 内核模式：`FPlayerFFServiceKernel.exe`

说明：

- 发布态 UI 由 Electron 主进程内置编排拉起 ZLM + gateway（不需要手动执行 `start-all.ps1`）
- UI 模式会显示启动状态，建议等待“服务已就绪”后再让 desktop 发起推流/拉流解析
- 内核模式为独立控制台启动器（Go），命令行会持续打印状态与端点信息，支持 `Ctrl+C` 停止
- 实际服务地址仍以 `run/runtime.json` 中 `gateway.url` 为准

## 4. 命令行停止（Windows）

```powershell
.\scripts\stop-all.ps1
```

会按 pid 文件停止 UI、gateway、ZLM，并清理 `run/runtime.json`。

## 5. desktop 如何填写“服务端模式”

在 desktop 推流窗口选择 `推送服务端` 后填写：

- 服务地址：`run/runtime.json` 中 `gateway.url`
- 服务 app：例如 `live`
- 服务 stream：例如 `stream001`
- 服务类型：
  - `broadcast`：广播分发（RTMP + FLV + HLS）
  - `httpflv`：广播分发 + HTTP-FLV 优先（含 HLS）

注意：

- 服务地址就是 service 机器地址，不是 desktop 本机地址。
- 若跨机器部署，请放行 service 机器防火墙端口：
  - `run/runtime.json` 中 `gateway.port`（默认优先 `9000`）
  - `run/runtime.json` 中对应的 RTMP/HTTP-FLV 端口

## 6. API 简要

### 6.1 创建会话

- `POST /api/v1/streams/start`

示例 body：

```json
{
  "app": "live",
  "stream": "stream001",
  "serviceMode": "httpflv",
  "publisherMeta": {
    "publisherIp": "192.168.1.23",
    "publisherPort": 1935,
    "scene": "camera"
  },
  "sourceMeta": {
    "fps": 30,
    "bitrateKbps": 4000,
    "videoEncoder": "auto",
    "audioInput": "off",
    "audioOutput": "system"
  }
}
```

返回字段重点：

- `publishRtmp`
- `playHttpFlv`
- `playUrls`
- `id`（streamId）

### 6.2 查询会话

- `GET /api/v1/streams/{id}/status`

可用于拉流端按 `streamId` 自动解析可拉流地址。

## 7. 常见问题

### 7.1 UI 报 `Failed to fetch`

优先检查：

- `http://<service-ip>:<gateway-port>/healthz` 是否可达（`gateway-port` 以 `run/runtime.json` 为准）
- `gateway` 是否运行（看 `logs/gateway.err.log`）
- desktop 填写的服务地址是否正确（是否误填 `127.0.0.1`）

### 7.2 看不到 UI 窗口

检查：

- `ui/node_modules` 是否安装完整
- `logs/ui.err.log` 是否有 `electron` 命令缺失或下载失败

### 7.3 端口冲突

`start-all.ps1` 已做自动端口选择，实际端口以 `run/runtime.json` 为准。

## 公网部署教程

本节用于将 service 部署到公网服务器（云主机/IDC），并为公网客户端提供推流与拉流能力。

### 1. 先确认端口策略（必需/可选）

必需暴露（至少）：

- `gateway.port`（默认优先 `9000`，以 `run/runtime.json` 为准）
- `zlm.rtmpPort`（默认优先 `1935`，以 `run/runtime.json` 为准）
- `zlm.httpPort`（默认优先 `8080`，以 `run/runtime.json` 为准）

按需暴露：

- `zlm.httpsPort`（默认优先 `8443`，如需 HTTPS 播放）
- `zlm.rtspPort`（Linux 默认优先 `8554`，Windows 常见 `554`；以 `run/runtime.json` 为准）
- WebRTC 端口段（如启用 WebRTC，再按 ZLM 配置放行 UDP 端口段）

### 2. 为什么不能只放行 9000

- `9000` 仅是控制面（创建会话、查询状态）。
- 实际媒体数据面走 `publishRtmp`、`playHttpFlv`、`playHls` 对应端口。
- 所以必须同时放行 gateway + ZLM 实际媒体端口。

### 3. 公网发布推荐步骤

1) 启动 service（推荐发布包）  
2) 读取 `run/runtime.json` 获取本次实际端口  
3) 在云安全组与服务器防火墙同时放行这些端口  
4) 从公网客户端访问 `http://<public-ip>:<gateway-port>/healthz` 验证  
5) 在 UI 创建流并使用返回的 `publishRtmp` / `playHttpFlv` / `playHls` 验证

### 4. Linux 防火墙示例（UFW）

以下示例仅供参考，请按 `run/runtime.json` 实际端口替换：

```bash
sudo ufw allow 9000/tcp
sudo ufw allow 1935/tcp
sudo ufw allow 8080/tcp
sudo ufw allow 8443/tcp
sudo ufw allow 8554/tcp
sudo ufw reload
```

### 5. 公网部署注意事项

- 对公网场景，建议固定公网 IP 或绑定域名，避免客户端地址漂移。
- 跨公网时不要使用 `127.0.0.1` 或内网地址。
- 若你使用了动态端口策略，务必以 `run/runtime.json` 为准同步安全组与客户端配置。
- 建议启用访问控制（例如反向代理鉴权、IP 白名单），避免服务裸露在公网被滥用。

### 6. Linux 端口权限（root / 非 root）

Linux 与 Windows 的端口权限策略不同，需特别注意：

- Windows：普通用户通常可直接绑定常见媒体端口（如 `554`）。
- Linux：普通用户默认不能绑定 `1024` 以下端口（如 `554`），会出现 `permission denied`。

当前项目策略：

- Linux 非 root 运行时，`rtsp` 使用高位端口（默认优先 `8554`，实际以 `run/runtime.json` 为准）。
- `rtmp/http/gateway` 仍按可用端口策略启动（常见为 `1935/8080/9000`，冲突时自动切换）。

若你必须在 Linux 使用低位端口（例如固定 `554`）：

1. 以 root 运行服务（不推荐作为默认方式）；或  
2. 对可执行文件授予绑定低位端口能力（更推荐）：

```bash
sudo setcap 'cap_net_bind_service=+ep' /path/to/MediaServer
getcap /path/to/MediaServer
```

说明：

- 使用 `setcap` 后通常无需整进程提权为 root。
- 升级/替换 `MediaServer` 二进制后，能力位可能丢失，需要重新执行 `setcap`。

### 7. systemd 开机自启与长期运行（推荐）

本节用于 Linux 服务器长期运行场景（开机自启、异常自动拉起、统一日志）。

#### 7.1 准备建议

- 使用发布目录运行（推荐）：`release/portable-ui` 或 `release/portable-kernel`
- 建议专用系统用户运行（如 `ffservice`），避免使用 root 长期运行
- 若需要低位端口（如 `554`），优先使用 `setcap`，不要默认 root

#### 7.2 service 文件示例（内核模式）

以下示例以 `portable-kernel` 为例，适合无界面服务器：

```ini
[Unit]
Description=FPlayer FF Service Kernel
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=ffservice # 删除，或者改成你当前用户
Group=ffservice # 删除，或者改成所属用户组
WorkingDirectory=/opt/fplayer-ff-service/release/portable-kernel # 改成应用所在绝对目录的父目录
ExecStart=/opt/fplayer-ff-service/release/portable-kernel/FPlayerFFServiceKernel # 改成应用所在绝对目录
Restart=always
RestartSec=2
LimitNOFILE=1048576
Environment=SERVICE_MANAGED_MODE=1

# 若确需绑定低位端口，可启用以下能力（需结合实际安全策略评估）
# AmbientCapabilities=CAP_NET_BIND_SERVICE
# CapabilityBoundingSet=CAP_NET_BIND_SERVICE
# NoNewPrivileges=true

[Install]
WantedBy=multi-user.target
```

保存为：

- `/etc/systemd/system/fplayer-ff-service.service`

#### 7.3 启用与管理

```bash
sudo systemctl daemon-reload
sudo systemctl enable fplayer-ff-service
sudo systemctl start fplayer-ff-service
sudo systemctl status fplayer-ff-service
```

常用操作：

```bash
sudo systemctl restart fplayer-ff-service
sudo systemctl stop fplayer-ff-service
sudo systemctl disable fplayer-ff-service
```

#### 7.4 日志查看与排障

```bash
# 实时看日志
journalctl -u fplayer-ff-service -f

# 最近 200 行
journalctl -u fplayer-ff-service -n 200 --no-pager
```

同时可结合应用目录内日志：

- `logs/zlm.log`
- `logs/gateway.log`
- `logs/*.err.log`

#### 7.5 更新发布包后的重启流程

1. 替换发布目录文件  
2. 若替换了 `MediaServer` 且使用了 `setcap`，重新执行能力授权  
3. 重启服务：

```bash
sudo systemctl restart fplayer-ff-service
```

#### 7.6 UI 模式的 systemd 说明

- `portable-ui` 依赖图形环境，不适合无头服务器长期后台运行。
- 服务器场景推荐 `portable-kernel` + API/日志管理。
- 若确需 UI 自启动，应基于用户会话（user service）并确保图形会话可用。
