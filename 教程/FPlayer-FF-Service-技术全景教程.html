01

项目概览

这个项目是做什么的

FPlayer FF Service 是一个独立部署的流媒体服务程序。它的核心工作只有三件事：

  1. 接收视频数据：接收来自采集端（桌面客户端）通过网络推送过来的音视频流
  2. 转换播放格式：把收到的同一份视频数据自动转成多种播放格式（RTMP、HTTP-FLV、HLS 等），让不同类型的播放端都能使用
  3. 管理播放地址：生成并维护推流地址和播放地址，让调用方不需要手动拼接 URL

它不采集画面，也不播放画面。它只负责"收 → 转 → 发"这个中间环节。它运行在 Windows 或 Linux 机器上，通过网络接口与其他程序通信。

在 FPlayer 生态中的位置

FPlayer FF Service 不是一个孤立软件。它是三个项目组成的生态系统中的"服务端枢纽"：

  • fplayer-ff-desktop（桌面客户端）：负责采集屏幕/摄像头画面，推流到 Service，也可以从 Service 拉流播放
  • fplayer-ff-service（本服务）：负责接收推流、协议转换、地址编排、流分发
  • fplayer-ff-mobile（手机客户端）：从 Service 获取播放地址，拉流观看

它们的典型使用场景是：同一局域网内，一台电脑采集并推流 → Service 中转分发 → 其他设备拉流观看。

项目由哪几部分组成

这个服务程序内部包含四个独立的程序组件：

02

流媒体基础概念

这一节不涉及具体代码，但每一个概念都会在后续模块中反复出现。这里把它们逐个讲清楚，追溯到各自依赖的前置知识。

"视频流"与"视频文件"的区别

你平时看本地视频——比如双击一个 .mp4 文件——播放器是把整个文件读入，然后逐帧解码播放。如果文件还没下载完，就播不了。

而"视频流"（stream）是一段持续产生、边产边传的数据序列。采集端不断产生新的画面帧和声音数据，通过网络持续发送出去；接收端持续接收并立即解码播放，中间不需要等一个"完整文件"。

直播必须用"流"——因为画面是实时产生的，不可能等录完一个文件再传。

前置知识：TCP 与 UDP

网络传输有两个最基本的协议——TCP（Transmission Control Protocol，传输控制协议）和 UDP（User Datagram Protocol，用户数据报协议）。理解它们的区别是理解后续所有流媒体协议的基础。

  • TCP —— 发送方每发一批数据，都会等接收方回复"收到了"，如果没收到就重发。所以 TCP 保证数据完整、顺序正确、不丢失。代价是需要往返确认，速度稍慢。
  • UDP —— 发送方只管发，不确认对方是否收到。所以 UDP 速度快但可能丢数据、可能乱序。

什么场景用哪个？文件传输、网页浏览、电子邮件——必须用 TCP（数据一点不能错）。视频通话、在线游戏——通常用 UDP（丢几帧无所谓，但延迟必须低）。流媒体推流通常用 TCP（画面完整性优先），实时通话通常用 UDP（低延迟优先）。

推流协议：RTMP

RTMP（Real-Time Messaging Protocol，实时消息传输协议）由 Adobe（原 Macromedia）设计，最初用于 Flash Player 和服务器之间的音视频传输。

RTMP 基于 TCP，所以数据不会丢。它的交互模式是"客户端主动连接到服务器，建立持久连接，然后在这个连接上持续推送音视频数据"。所以叫"推"流——客户端把数据推向服务器。

RTMP 的默认端口号是 1935。"端口号"可以理解为操作系统给每个网络程序分配的数字标识——操作系统收到网络数据后，根据端口号判断该把数据交给哪个程序。

在这个项目里，桌面客户端（Desktop）用 RTMP 把采集到的视频画面推送到 Service 的 ZLMediaKit。

拉流协议一：HTTP-FLV

FLV（Flash Video）是一种视频容器格式——它规定了音视频数据怎么打包到一个文件里。HTTP-FLV 的意思是：通过 HTTP 协议来传输 FLV 格式的视频流。

这里需要先理解 HTTP：HTTP（HyperText Transfer Protocol）就是你浏览器访问网页用的协议。你在地址栏输入 URL，浏览器向服务器发一个 HTTP 请求，服务器返回网页内容。HTTP 基于 TCP，所以也是可靠的。

HTTP-FLV 的做法是：播放端向服务器发一个 HTTP 请求，服务器不立即结束响应，而是持续不断地在响应体里追加 FLV 数据。播放端收到一段播一段。因为数据是持续推送的（不需要等"切片"），所以延迟很低，通常在 1-3 秒。

缺点是部分手机浏览器（特别是 iOS Safari）对 FLV 格式支持不好，需要借助额外的播放库。

拉流协议二：HLS

HLS（HTTP Live Streaming）是苹果公司 2009 年推出的流媒体协议。它的工作原理和 HTTP-FLV 完全不同：

  1. 服务器把视频流切成一个个短小的 .ts 文件（每个文件通常 2 到 10 秒的内容，本项目是 2 秒）
  2. 同时维护一个索引文件（扩展名 .m3u8），里面列出了当前有哪些 .ts 片段、它们的顺序、以及播放地址
  3. 播放端先下载 .m3u8 索引，然后按顺序逐个请求并播放这些 .ts 片段

HLS 的特点：

  • 兼容性最好 —— iOS、Android、所有主流浏览器都原生支持，不需要额外插件
  • 延迟较高（通常 5-10 秒）—— 因为必须等一个片段完全切完、封好文件，播放端才能下载到它
  • 容错性好 —— 如果某一片段下载慢了，不影响已缓冲的后续片段；也可以随时切换到不同画质的流（自适应码率）

在这套系统中，播放端可以根据场景选协议：需要低延迟用 HTTP-FLV，需要兼容性用 HLS。

概念回顾

RTMP

03

ZLMediaKit 媒体引擎

它是什么

ZLMediaKit 是一个用 C++ 编写的高性能流媒体服务器程序，开源在 GitHub 上。在本项目中，它被作为预编译的二进制文件直接使用——也就是别人已经编译好的可执行程序，放在 3rd/zlm/ 目录下（Windows 版是 MediaServer.exe，Linux 版是 MediaServer）。项目不需要重新编译它。

ZLMediaKit 支持非常多的流媒体协议：RTMP、RTSP、HLS、HTTP-FLV、WebRTC、SRT、TS、fMP4 等。但在这个项目中，主要用到的是 RTMP（接收推流）和 HTTP-FLV / HLS（对外分发）。

协议转换是怎么做的

首先要理解一个重要区分：视频编码和传输协议是两回事。

视频编码（如 H.264、H.265）决定画面怎么被压缩成数据。传输协议（如 RTMP、HLS）决定数据怎么被打包、走什么网络通道到达播放端。同一份 H.264 编码的视频数据，可以装在 RTMP 的包里传输，也可以切成 HLS 的 .ts 片段传输，也可以封装成 FLV 格式通过 HTTP 传输。

协议转换——在流媒体领域更准确的叫法是"转封装"——就是：把通过协议 A 收到的音视频数据，解出里面的编码数据，然后用协议 B 的格式重新打包，通过协议 B 的通道发出去。不改变编码本身，只改变封装和传输方式。

ZLMediaKit 在收到 RTMP 推流后，会自动、实时地生成以下输出：

  • RTMP 输出（供支持 RTMP 的播放器拉流）
  • HTTP-FLV 输出（供网页端、桌面端播放）
  • HLS 输出（.m3u8 + .ts 片段，供手机端播放）

这一切是自动完成的——推流端只管推 RTMP，ZLMediaKit 自动在后台完成所有协议的转封装。

关键配置：关闭"按需"

ZLMediaKit 有一个配置项叫"按需拉流"（on-demand）。如果开启这个选项，ZLMediaKit 只有在有播放端请求某个协议时才会开始生成该协议的输出。如果没人看，就不生成。省计算资源。

这个项目选择关闭所有协议的按需——推流一到达，所有协议输出立刻开始生成，不管有没有人在看。代价是始终占用 CPU 做转封装。好处是首帧播放延迟极低：第一个播放端请求到达时，数据已经准备好了，不需要等待转换启动。

HLS 的切片参数也为此做了优化：每片 2 秒（常规是 6-10 秒），保留最近 5 个分片，保留 3 个在线。切片越短，播放端从"请求播放"到"看到画面"的等待时间越短。

# ZLMediaKit config.ini — 关键配置 [protocol] enable_hls=1 enable_rtmp=1 enable_rtsp=1 enable_fmp4=1 hls_demand=0 # 关闭 HLS 按需 → 推流即转 rtmp_demand=0 # 关闭 RTMP 按需 [hls] segDur=2 # 切片时长 2 秒（低延迟） segNum=3 # 每个 m3u8 里保留 3 个切片 segRetain=5 # 磁盘上保留 5 个切片

Gateway API 网关

它是什么

Gateway 是一个用 Go 语言编写的 HTTP 服务程序。什么是 HTTP 服务？就是一段程序，它在某个端口上监听，收到 HTTP 请求后根据 URL 路径执行对应的逻辑，然后返回 HTTP 响应（通常是 JSON 格式的数据）。

Gateway 的特点：

  • 单文件：全部代码在 gateway/main.go 一个文件里，约 860 行
  • 零外部依赖：不依赖任何第三方 Go 库，只用 Go 语言自带的"标准库"（net/http、encoding/json、sync 等）
  • 编译后自包含：编译产物是一个独立的可执行文件，拷贝到任意同架构的机器上就能运行，不需要安装 Go 运行时

全部 API 端点

Gateway 通过以下 8 个 URL 路径对外提供服务（假设 Gateway 运行在 9000 端口）：

方法路径说明
GET/healthz健康检查。返回 {"status":"ok"}。所有启动器都通过轮询这个接口判断 Gateway 是否就绪
POST/api/v1/streams/start创建流。请求体传入 app、stream、serviceMode。同一个 app+stream 重复调用不报错，直接返回已有结果（幂等）
GET/api/v1/streams/resolve按名称查地址。URL 参数传 app 和 stream，不传流 ID。专为播放端设计——播放端只知道名称
GET/api/v1/streams/{id}/status查询指定流的状态，URL 路径里包含流 ID。返回状态、推流地址、各协议播放地址
POST/api/v1/streams/{id}/stop停止指定流。在 Gateway 内存中将其状态标记为 "stopped"
GET/api/v1/streams列出当前所有流会话
GET/api/v1/debug/logs查看 ZLM、Gateway、UI 的运行日志尾部内容。支持 limit 参数（最大 2000 行）。调试用
GET/api/v1/debug/network查看当前网络状态：主机地址、各端口号、公网代理状态、操作系统。调试用

三种服务模式

创建流时，serviceMode 参数决定 Gateway 生成哪些协议的播放地址：

direct

仅生成 RTMP 播放地址。最精简，适合纯 RTMP 播放场景。

RTMP only

broadcast

同时生成 RTMP + HTTP-FLV + HLS。覆盖所有主流播放方式。

全协议

httpflv

行为与 broadcast 一致（也生成 RTMP + HTTP-FLV + HLS）。名称是历史遗留，表示侧重网页端场景。

Web 友好

数据存储方式

Gateway 把所有流信息存在进程内存中：

// gateway/main.go — 核心数据结构 var ( mu sync.RWMutex // 读写锁 streams = map[string]*streamRecord{} // 键=流ID, 值=流信息 )

用 map（映射表/字典）存储，以随机生成的 16 位十六进制字符串作为键（即流 ID）。

sync.RWMutex 是一个读写锁——它解决的是"多个请求同时操作同一份数据"的并发安全问题：读操作可以多个同时进行（不会互相干扰），写操作必须独占（写的时候其他人不能读也不能写）。

这意味着Gateway 重启后所有流数据都会丢失。这是有意为之——参见第 12 节设计决策。

"幂等"是什么意思

在计算机科学中，幂等（Idempotent）指：同一个操作执行一次和执行多次，结果相同。

Gateway 的 POST /api/v1/streams/start 接口是幂等的。如果你两次用相同的 app + stream 参数调用它：第一次正常创建并返回流信息；第二次不会报错，也不会创建新流——直接在内存中找到第一次创建的记录，返回同样的信息。

这样设计的好处是：调用方（Desktop）不需要自己判断"这个流是不是已经创建过了"。只管调，Gateway 自己处理重复的情况。

05

流会话管理

一次完整的流会话长什么样

在 Gateway 里，一次视频流的全过程被抽象为一个"会话"（session）。从创建到停止，涉及四个环节：

环节一：创建会话

Desktop 向 Gateway 发 POST /api/v1/streams/start，传入 app（应用名）、stream（流名）、serviceMode（模式）。Gateway 先做幂等检查——如果已有同名运行中的流，直接返回已有记录。否则：生成 16 位随机 ID → 创建流记录 → 拼推流和播放地址 → 存入内存 → 返回 JSON。

环节二：推流

Desktop 拿到返回的 publishRtmp 地址，通过 RTMP 协议把视频数据持续推送到 ZLMediaKit。ZLMediaKit 实时转封装出 HTTP-FLV 和 HLS 输出。

环节三：查询与拉流

播放端（Mobile 或另一个 Desktop）向 Gateway 发 GET /api/v1/streams/resolve?app=live&stream=stream001。Gateway 在内存中找到对应记录，返回播放地址。播放端用这些地址向 ZLMediaKit 请求视频数据并播放。

环节四：停止

推流端调用 POST /api/v1/streams/{id}/stop。Gateway 把该记录的状态标记为 "stopped"。注意：这只是修改了 Gateway 里的标记，ZLMediaKit 中的流生命周期由 ZLMediaKit 自己管理。

流会话生命周期：创建 → 推流 → 转封装 → 拉流播放。①③走 Gateway（控制面），②④走 ZLMediaKit（数据面）

Electron 管理界面

Electron 是什么

要理解 Electron UI，需要先理解 Electron 框架本身。Electron 是一个开源框架，让你能用写网页的技术（HTML、CSS、JavaScript）来构建桌面窗口应用。

它本质上由两部分组成：

  • Chromium：Google Chrome 浏览器的开源内核，负责渲染网页界面、执行 JavaScript。你把 HTML 和 CSS 写好，Chromium 负责把它们显示成窗口内容
  • Node.js：一个让 JavaScript 脱离浏览器、在操作系统层面运行的运行时。它能读写本地文件、启动子进程、访问网络——普通浏览器里的 JavaScript 是做不到这些的（出于安全限制）

Electron 把 Chromium 和 Node.js 嵌在一个可执行程序里。界面用网页技术写（简单），系统操作用 Node.js 做（能力强）。两者通过 IPC（Inter-Process Communication，进程间通信）协作。

这个项目里的 UI 做了什么

  • 流管理面板：输入 Gateway 地址、app、stream、模式 → 创建流 → 显示返回的推流/播放地址 → 一键复制
  • 日志查看器：四个日志区域（Gateway 标准输出、Gateway 错误、ZLMediaKit、UI 错误），每 3 秒自动刷新
  • 按需启动：窗口打开时不自动启动 ZLMediaKit 和 Gateway。用户点击"创建并开始"按钮才启动——避免只是打开看看就占用了端口
  • 深浅色主题：右上角切换，偏好保存在 localStorage

两种运行模式

  • 托管模式（发布版）：Electron 的 main.js（主进程）直接启动和管理 ZLMediaKit + Gateway 作为子进程。包括端口探测、配置生成、健康检查、写入 runtime.json——整套启动逻辑在 JavaScript 里实现了一遍
  • 脚本模式（开发版）：依赖外部的 scripts/start-all.ps1（Windows）或 start-all.sh（Linux）来启动服务。Electron 本身只做界面展示

打包时，通过 electron-builder 把 ZLMediaKit 二进制、Gateway 编译产物、启动脚本一并打包进 extraResources。最终输出的 .exe（Windows）或 AppImage（Linux）包含了运行需要的全部文件。

07

Kernel Console 启动器

它是什么

Kernel Console 是一个用 Go 编写的命令行程序，约 650 行，以 FPlayerFFServiceKernel.exe（Windows）或 FPlayerFFServiceKernel（Linux）作为独立可执行文件分发。

它的使用场景是不需要图形界面的环境——比如远程服务器、NAS、树莓派等。直接在终端里运行，启动后前台驻留，打印运行信息，按 Ctrl+C 停止。

它做了什么

  1. 自动探测目录：从自身可执行文件的位置出发，向上查找服务根目录（通过检查 3rd/zlm/、gateway/bin/、scripts/ 等标志性文件是否存在来判断）
  2. 动态分配端口：为 ZLMediaKit 的 RTMP、HTTP、HTTPS、RTSP（仅 Linux）端分别探测空闲端口
  3. 生成配置：读取 ZLMediaKit 的配置模板，替换端口号，写入运行时配置 run/zlm.runtime.ini
  4. 启动子进程：先启动 ZLMediaKit，通过 TCP 连接到 RTMP 端口验证其就绪；再启动 Gateway，轮询 /healthz 直到返回 200
  5. 自动创建流：Gateway 启动后，调用 POST /api/v1/streams/start 创建一个默认流（app=live, stream=stream001, mode=httpflv）
  6. 打印运行摘要：在终端输出所有关键信息——Gateway URL、健康检查地址、各协议推拉流地址——格式对齐，方便复制
  7. 前台驻留与优雅停止：监听 SIGINT/SIGTERM 信号，收到后先发中断信号给子进程、等 1.5 秒、超时则强制终止

服务器部署

三套实现的启动步骤完全一致，但代码各自独立编写（用不同语言），不互相调用。这样做是为了避免"跨语言依赖链"——如果脚本依赖 Go 程序、Electron 依赖脚本，任何一环改动都可能影响其他启动路径。各自独立实现，虽然代码有重复，但每条路径都完全自包含。

启动需要哪几步

  1. 端口探测：依次尝试默认端口（RTMP=1935, HTTP=8080, Gateway=9000 等），如果被占用就用操作系统随机分配的空闲端口
  2. 配置生成：读取 ZLMediaKit 的模板配置文件，把端口号写进去，生成 run/zlm.runtime.ini
  3. 启动 ZLMediaKit：作为子进程启动，把标准输出和标准错误重定向到日志文件
  4. 验证 ZLMediaKit：通过 TCP 连接探测 RTMP 端口，确认 ZLMediaKit 已就绪（Electron 模式超时 10 秒）
  5. 启动 Gateway：作为子进程启动，注入环境变量（ZLM 端口号、Gateway 端口号等）
  6. 健康检查轮询：每 200 毫秒请求一次 /healthz，最多等 5 秒。如果超时，换一个空闲端口重试，最多 8 次
  7. 写入运行时信息：将 Gateway URL、各端口号、启动时间等写入 run/runtime.json

端口动态分配

一台电脑有 65536 个端口（0-65535）。每个网络程序监听其中若干端口。问题是：一个端口同一时刻只能被一个程序占用。如果默认端口被其他程序占了，服务就会启动失败。

解决方案——先试默认端口，被占就随机分：

// kernel-console/main.go — 端口分配 func getFreePort(preferred int) int { if preferred > 0 { if p := try(preferred); p > 0 { return p } // 先试默认值 } return try(0) // 被占了就让操作系统随机给一个 }

Gateway 更加顽强——启动后如果 /healthz 没有在 5 秒内返回成功，自动换端口重试，最多 8 次。

优雅停止

停止服务不能直接"杀进程"。正确的做法叫"优雅停止"（graceful shutdown）：

  1. 先向子进程发送 SIGINT 信号（等同于在终端按 Ctrl+C）。这是通知："请你正常退出"
  2. 等待 1.5 秒，让子进程有机会清理资源、关闭网络连接、写入未完成的数据
  3. 如果 1.5 秒后还没退出，发送 SIGKILL 信号——操作系统直接终止进程，不给任何反应时间

停止顺序：先停 Gateway，再停 ZLMediaKit。Gateway 依赖 ZLMediaKit（需要知道端口信息），但 ZLMediaKit 不依赖 Gateway。先关依赖方，后关被依赖方，逻辑上不出错。

网络与地址编排

为什么 IP 地址选择是个问题

Gateway 需要把正确的 IP 地址写入播放 URL（比如 rtmp://192.168.1.5:1935/live/stream001）。但一台电脑可能同时有多个 IP 地址：

  • 有线网卡一个 IP（如 192.168.1.10）
  • 无线网卡一个 IP（如 192.168.1.11）
  • VPN、Docker、虚拟机等虚拟网卡也会创建 IP
  • 127.0.0.1（回环地址）——只能本机访问，局域网内其他设备用不了

如果 Gateway 选了一个错误的 IP——比如选了 127.0.0.1——那么同一局域网内的手机就无法访问。所以需要一套策略来选对的 IP。

Gateway 的多级 IP 选择策略

Gateway 按以下优先级选择发布地址：

  1. 公网配置覆盖：如果服务启动时设置了 SERVICE_PUBLIC_HOST，直接使用这个地址（适用于通过 FRP 等工具做了公网映射的场景）
  2. 启动阶段选定：Gateway 启动时遍历所有可用 IP，做绑定可达性测试，选第一个能用的，记录在 selectedPublishHost 变量中
  3. 请求阶段动态选择（Linux 下更精细）：
    • 路由优选——通过 UDP 拨号探测从客户端过来的网络路径走的是哪个网卡，选那个网卡的 IP
    • 同 /24 子网优先——如果客户端 IP 是 192.168.1.x，优先选同样是 192.168.1.y 的服务器 IP（IP 前三位相同意味着在同一个广播域内）
    • 轮询可达性——逐个对候选 IP 做 TCP 连接测试（300 毫秒超时）
  4. 请求回退：取 HTTP 请求头中的 X-Forwarded-Host 或 Host；都没有就用 localIPv4()；再没有就用 127.0.0.1

CORS 中间件

Gateway 有一个 corsMiddleware 函数。要理解它为什么存在，需要先理解浏览器的同源策略（Same-Origin Policy）。

同源策略是浏览器的一个安全机制：A 网站（http://example.com）上的 JavaScript 默认不能向 B 网站（http://api.example.com）发送 HTTP 请求——即使 B 网站的拥有者是你自己。浏览器会拦截这个请求。这是为了防止恶意网站冒用你的登录信息去访问其他网站。

但实际开发中，前后端分离很常见——网页界面在一个域名，API 在另一个域名。这时就需要 CORS（Cross-Origin Resource Sharing，跨域资源共享）：服务端在 HTTP 响应头里加上标记，告诉浏览器"我允许跨域访问"。

Gateway 的 CORS 设置：

  • Access-Control-Allow-Origin: * —— 允许任意来源的请求
  • Access-Control-Allow-Headers: Content-Type —— 允许请求方声明内容格式
  • Access-Control-Allow-Methods: GET,POST,OPTIONS —— 允许的方法
  • 对 OPTIONS 预检请求直接返回 204，不做实际处理

设置为 *（任意来源）是最宽松的配置。在当前版本中没有鉴权机制，所有接口都是公开的，所以这样配置是合适的——反正没有敏感数据需要保护。

三端协同

FPlayer FF Service 不是孤立存在的。它和另外两个项目共同组成一个完整系统：

三个项目的协作模式：

  1. Desktop 把采集到的视频画面通过 RTMP 推送到 Service 的 ZLMediaKit
  2. Service 的 ZLMediaKit 自动转封装为 HTTP-FLV 和 HLS；Gateway 维护流会话并生成播放地址
  3. Mobile（或其他 Desktop）通过 Gateway 的 resolve 接口，用 app + stream 名称查到播放地址，直接拉流播放

注意：Mobile 不需要知道流 ID，不需要知道端口号，不需要知道协议细节。它只需要知道应用名（app）和流名（stream）。所有复杂的地址拼接都在 Gateway 端完成。这就是"地址编排"的价值所在。

构建与分发

Go 程序的编译

Go 语言的一个重要特性：编译为独立二进制文件。编译产物是一个可执行文件，包含了 Go 运行时和所有依赖，不需要在目标机器上安装 Go。

编译命令：

# Windows go build -o gateway/bin/gateway.exe gateway/ go build -o kernel-console/bin/FPlayerFFServiceKernel.exe kernel-console/ # Linux go build -o gateway/bin/gateway gateway/ go build -o kernel-console/bin/FPlayerFFServiceKernel kernel-console/

Electron 打包

Electron 应用通过 electron-builder 打包。它会：

  1. 把 HTML/CSS/JS 界面资源打包
  2. 嵌入 Chromium 引擎和 Node.js 运行时
  3. 通过 extraResources 配置把外部文件（ZLMediaKit 二进制、Gateway 编译产物、启动脚本）一并打进包
  4. Windows 上输出独立的 .exe 可执行文件（portable 模式，无需安装）
  5. Linux 上输出 AppImage 格式

最终产物的拆分

构建脚本最终生成两个独立分发包：

  • portable-ui：含 Electron 图形界面 + ZLMediaKit + Gateway + 脚本。面向需要可视化管理的用户
  • portable-kernel：含 Kernel Console + ZLMediaKit + Gateway + 脚本。面向命令行/服务器部署场景

两个包共用底层的 ZLMediaKit 和 Gateway 二进制文件，只是上层入口不同。各自的构建脚本还会做完整性校验——检查所有必需文件是否存在，避免打包遗漏。

跨平台适配

Kernel Console 启动时会自动检测操作系统（runtime.GOOS），然后选择正确的可执行文件名（Windows 加 .exe 后缀）、正确的 ZLMediaKit 目录（Windows 用 3rd/zlm/windows/，Linux 兼容 linux 和 Linux 两种大小写）、以及适当的端口策略（Linux 额外开启 RTSP 端口，默认 8554 以避免需要 root 权限绑定 554）。

设计决策

不做持久化数据库

Gateway 的流信息全部存在进程内存里，重启丢失。这是有意的设计取舍：

  • 视频流本身就是临时的——直播结束就没了。存储历史记录的意义有限
  • 消除数据库依赖——不需要安装 MySQL/Redis，不需要维护表结构。Gateway 就是个可执行文件，拷贝即部署
  • 如果未来需要历史记录，可以在 Gateway 上层再做一层服务——调用 Gateway API 获取流信息后存到自己的数据库里。Gateway 本身保持简单

不做强鉴权

当前版本的 API 没有任何身份验证机制——谁都可以调 POST /api/v1/streams/start 创建流。原因：

  • 当前使用场景是同一局域网内的可信环境——所有设备都在同一个内网，没有外部攻击面
  • 如果需要公网部署，可以通过网络层的防火墙/VPN 解决安全问题
  • 未来可以加 publisher token（推流令牌）+ viewer token（观看令牌），但当前不是必需的

不做转码

ZLMediaKit 只做协议转换（转封装），不做编码转换（转码）。区别：

  • 转封装：H.264 数据用 RTMP 收 → 用 HLS 发。编码不变，只换包装
  • 转码：H.265 数据 → 解码 → 重新编码为 H.264。需要大量 CPU

不做转码意味着推流端用什么编码，播放端就得支持什么编码。这简化了服务端的计算负担——所有计算压力在推流端和播放端，服务端只是个高效的中转站。

Gateway 为什么只有一个文件

860 行 Go 代码，全在 main.go 里，没有拆成多个包。这不是偷懒——Go 社区有很多成功的单文件 HTTP 服务案例。当程序的功能范围明确、代码量不大时，拆成多个文件反而增加了阅读和跳转的成本。一个文件，从上往下读，逻辑自然展开。

零第三方依赖也是同理——Go 标准库已经提供了 net/http（HTTP 服务）、encoding/json（JSON 编解码）、sync（并发控制）、crypto/rand（随机数），这些足够完成 Gateway 需要的所有功能。不引入框架，就没有框架升级带来的兼容性问题。
